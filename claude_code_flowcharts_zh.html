Claude Code 架构与执行流程图

本文档已按当前主报告的已验收表述同步，重点修正宿主关系、Prompt / Context / Cache 主线、安全边界拆层，以及 Session 恢复与治理闭环。

1. 一级边界与主信息流

这张图表达的是 Claude Code 的一级边界和主信息流，不是源码里一条严格单向的调用栈。关键点是：REPL 与 Headless 是并列宿主，二者都把请求送进共享 query() 内核；Prompt / Context / Cache 在进入 query() 之前就已经成立；Tool 安全链、MCP、Bridge / Remote Control、Session 治理是相邻边界，不该再混成一层。

2. 启动分流与宿主装配

真正的启动链路不是“进入 CLI 后直接打模型”。入口先做快速路径分流和初始化，再决定进入交互式宿主、headless 宿主或特定子命令。REPL UI 和 QueryEngine 在这里是两种宿主壳，而不是上下级关系。

轻量分发器] Entry --> Fast{是否命中快速路径} Fast -- 是 --> FastHandlers[daemon / remote / bridge 等快速路径] Fast -- 否 --> Main[src/main.tsx
主启动编排] Main --> Setup[src/setup.ts
初始化上下文与服务] Setup --> Config[配置 / policy / auth / telemetry] Setup --> Tooling[工具注册 / MCP 客户端 / commands] Config --> HostChoice{进入哪类宿主} Tooling --> HostChoice HostChoice -- 交互式 --> ReplHost[src/replLauncher.tsx / REPL.tsx] HostChoice -- Headless --> HeadlessHost[src/QueryEngine.ts] HostChoice -- 子命令 --> Commands[src/commands/*] ReplHost --> HostState[宿主状态装配
resume / tools / settings / session state] HeadlessHost --> HostState HostState --> PromptPrefix[Prompt / Context / Cache Prefix] PromptPrefix --> QueryCore[共享 query() 内核] Commands --> End([进程退出]) FastHandlers --> End QueryCore --> End

3. 宿主与共享 query() 内核

这一节专门纠正旧版 “REPL -> QueryEngine” 误导。更准确的关系是：REPL 宿主承接交互、resume、UI 状态；Headless 宿主由 QueryEngine 承接程序化会话；二者都先完成宿主装配，再进入共享 query() 内核。

buildEffectiveSystemPrompt()] PromptLine --> CachePrefix[userContext / systemContext / cache-safe prefix] CachePrefix --> Query[共享 query() 主循环] Query --> Output[按宿主协议输出结果]

4. Tool 安全链、MCP 与 Bridge / Remote Control

这里故意把三件事拆开：Permission / Classifier / Sandbox 是本地执行安全链；MCP client / server 是外部能力接入层；Bridge / Remote Control 接的是宿主与 session 控制面，不属于 tool 执行层。

OAuth / session handle] -.接入宿主而非工具层.-> H

5. Session / Transcript / Memory / Compaction 闭环

这里不再使用 “load/save history” 那种过度简化的旧图。更准确的说法是：Session 是可恢复运行单元，Transcript 是可继续喂给 query() 的消息链，SessionMemory 是后台节流提炼出的长期事实资产，Compaction 是上下文治理闭环的一部分，并会反过来影响后续恢复与 cache 命中。

7. 上下文治理与压缩

上下文治理不是“只要超长就做一次摘要”。它是分层次的预算与压缩系统：先管 tool result 预算，再做 snip、microcompact、collapse，最后才进入 autocompact；必要时还会在错误后走 reactive compact。
flowchart TD A[开始上下文治理] --> B[tool result budget / content replacement] B --> C[snipCompactIfNeeded] C --> D[microcompact] D --> E[context collapse] E --> F[autocompact] F --> G[必要时 reactive compact] G --> H[继续 query loop]

8. 工具并发批次

工具并发也不是“所有只读工具一次性全开”。系统会根据 isConcurrencySafe 把工具拆成安全批次；可并发的工具同批执行，不安全的工具会形成串行屏障。
