# 《Agentic Design Patterns》记忆脑图内容源

> 用途：作为 `mindmap.html` 的结构化内容源说明。组织方式不是摘要页，而是“中心主题 → 4 个 Part → 21 章 → 核心组件 / 关键技术节点 / 记忆锚点”的记忆骨架。

## 中心主题

- 主题：`Agentic Design Patterns`
- 主线：从基础工作流，到状态与学习，再到可靠性与接地，最后进入协议、优化与高级智能。
- 记忆总口诀：`先把流程跑通，再把状态管住；先把系统做稳，再把智能做深。`

## Part 1：基础工作流与协作（第 1-7 章）

### 第 1 章：提示词链
- 章节定义：把复杂任务拆成多个顺序步骤，让上一步输出成为下一步输入，用链式结构替代“一次性大提示”。
- 核心组件：
  - 任务拆分器：把大目标切成若干可控小步。
  - 提示契约：为每一步定义输入、角色、输出格式。
  - 中间结果载体：把步骤产物稳定传递给下游。
  - 校验关口：在进入下一步前检查格式和质量。
- 关键技术节点：
  - 顺序执行而非一次生成。
  - JSON / XML / 表格等结构化输出。
  - 中间变量传递与上下文拼装。
  - 步骤级日志与断点检查。
  - 链尾可接工具、反思或路由。
- 典型落地：`抽取信息 → 转结构化数据 → 生成报告 / 邮件 / 代码说明`。
- 风险边界：链越长，错误传播越明显；步骤切太碎会让编排复杂度飙升。
- 记忆口诀：`先拆后传，步步验算。`

### 第 2 章：路由
- 章节定义：根据输入意图、语义或规则，把请求送到最合适的分支、工具或子 Agent。
- 核心组件：
  - 路由器：负责“该走哪条路”的决策。
  - 标签集合：稳定、有限、可解释的分类结果。
  - 分支处理器：每条路径的专用执行单元。
  - 兜底分支：处理低置信度、模糊输入与澄清。
- 关键技术节点：
  - 意图分类与标签输出。
  - 向量语义路由。
  - 规则路由与条件分支。
  - 低置信度澄清机制。
  - 路由日志与混淆样本回放。
- 典型落地：`客服分流、代码助手分工、研究任务按查询类型派发`。
- 风险边界：路由错一次，后面整条链都可能建立在错前提上。
- 记忆口诀：`先判路，再出手。`

### 第 3 章：并行化
- 章节定义：把互不依赖的子任务同时执行，以换取更低总延迟和更高吞吐。
- 核心组件：
  - 子任务切分器：识别哪些任务可以并发。
  - 并行工作单元：独立运行的多个分支。
  - 汇聚器：收集并合并并行结果。
  - 超时与预算控制器：限制慢分支拖累整体。
- 关键技术节点：
  - fan-out / fan-in 并发模式。
  - Map-Reduce 式结果聚合。
  - 分支隔离与状态分槽。
  - 部分失败占位与重试。
  - 限流、超时、速率控制。
- 典型落地：`并行搜集新闻 / 财报 / 社媒，再统一综合`。
- 风险边界：表面独立、实际共享前提的任务，盲目并发会造成冲突答案。
- 记忆口诀：`能并不串，先分后合。`

### 第 4 章：反思
- 章节定义：把执行过程改造成“生成 → 评审 → 修正”的闭环，让初稿不是终稿。
- 核心组件：
  - 生成器：先给出第一版结果。
  - 审查器：按标准找缺陷、提修改点。
  - 缺陷清单：结构化记录问题与建议。
  - 停止条件：定义何时收敛、何时退出。
- 关键技术节点：
  - generator / reviewer 角色分离。
  - 结构化批评输出。
  - 多轮迭代修正。
  - 迭代上限控制。
  - 历史批评累积进下一轮上下文。
- 典型落地：`代码修复、文案润色、规划优化、长答案打磨`。
- 风险边界：若评审标准不清，系统会反复重写却不真正收敛。
- 记忆口诀：`先做后审，错了再改。`

### 第 5 章：工具使用（函数调用）
- 章节定义：让模型在需要真实数据、精确计算或外部动作时调用工具，而不是只靠语言猜测。
- 核心组件：
  - 工具定义：名称、参数、说明、返回值。
  - 决策器：判断何时该调用哪个工具。
  - 执行器：真正发起 API / DB / 代码运行。
  - 观察回灌器：把工具结果重新送回模型。
  - 权限边界：控制副作用和安全范围。
- 关键技术节点：
  - 函数调用 schema。
  - 参数校验与默认值处理。
  - 工具结果标准化。
  - 副作用确认与鉴权。
  - 工具错误重试与异常承接。
  - Agent as Tool 抽象。
- 典型落地：`查库存、执行脚本、发邮件、查天气、访问企业知识系统`。
- 风险边界：工具越强，副作用越大；如果权限模型混乱，真实业务会暴露风险。
- 记忆口诀：`模型定意，工具落地。`

### 第 6 章：规划
- 章节定义：在目标明确但路径未知时，先生成计划，再按计划执行或边执行边重规划。
- 核心组件：
  - 目标定义器：说明想达到的终态。
  - 计划图：把目标拆成子目标和顺序。
  - 执行器：逐步落地计划动作。
  - 重规划器：在环境变化时改路线。
  - 检查点：用于复盘当前进度与偏差。
- 关键技术节点：
  - 目标分解与依赖排序。
  - 状态建模与约束表达。
  - 中途信息回流后的重规划。
  - 长时任务状态持久化。
  - 暴露中间步骤供审计与修正。
- 典型落地：`深度研究、复杂采购、长链路客服诊断、项目型任务执行`。
- 风险边界：计划看起来合理，不代表执行一定可靠；过度规划会拖慢行动。
- 记忆口诀：`先定终，再排程。`

### 第 7 章：多 Agent 协作
- 章节定义：用多个角色清晰、边界明确的 Agent 共同完成复杂任务，而不是把所有能力压进一个 Agent。
- 核心组件：
  - 角色集合：研究员、执行者、评审者、管理者等。
  - 通信拓扑：顺序、并行、监督者、层级或网络结构。
  - 共享状态：统一字段、统一语义、统一交接格式。
  - 委派规则：谁可分派、谁只执行、谁可审查。
  - 审计轨迹：记录交接与决策链。
- 关键技术节点：
  - manager / specialist 分工。
  - Sequential / Parallel / Loop 编排。
  - AgentTool 与 Agent 嵌套调用。
  - 共享本体与状态键约定。
  - 交接契约与产物格式。
  - 管理者 Agent 日志化。
- 典型落地：`研究-写作-审校、生成-测试-文档、供应链多角色协同`。
- 风险边界：角色拆太细会让系统变成“踢皮球网络”，调试和通信成本陡增。
- 记忆口诀：`分工清，交接明。`

## Part 2：状态、学习与控制（第 8-11 章）

### 第 8 章：记忆管理
- 章节定义：让 Agent 不再每轮都“失忆”，同时管理会话内短期状态与跨会话长期知识。
- 核心组件：
  - Session：一条会话线程的容器。
  - State：会话中的动态草稿本。
  - Long-term Memory：持久存储的经验或知识。
  - 检索器：把长期记忆召回进当前上下文。
  - 命名空间：组织 user / app / temp 等不同作用域。
- 关键技术节点：
  - 短期记忆 vs 长期记忆分层。
  - session.state 键值化管理。
  - user: / app: / temp: 作用域。
  - 事件驱动状态更新。
  - 长对话摘要压缩。
  - 语义检索 + 元数据过滤。
- 典型落地：`记住用户偏好、跨多轮跟踪任务、复用过往成功策略`。
- 风险边界：记忆不等于真相，过期事实和错误偏好会污染后续决策。
- 记忆口诀：`短期保上下文，长期存经验。`

### 第 9 章：学习和适应
- 章节定义：让 Agent 因过去经验而改变策略、结构或代码，形成跨周期的持续改进。
- 核心组件：
  - 经验档案库：保存历史版本、轨迹与评分。
  - 评估器：外部度量“改得是否更好”。
  - 更新器：按经验修改策略 / 参数 / 代码。
  - 监督者：观察并在失控时中止。
  - 变异与选择机制：保留优解、淘汰劣解。
- 关键技术节点：
  - PPO 稳定策略更新。
  - DPO 基于偏好的直接优化。
  - 在线学习与增量适应。
  - 自我改进编码 Agent。
  - 进化搜索与候选版本竞争。
  - 隔离环境中的自修改执行。
- 典型落地：`个性化助手持续调优、编码 Agent 自我进化、推荐 / 交易系统在线适应`。
- 风险边界：如果评估器设计差，系统会学会“刷分”，而不是学会真正变好。
- 记忆口诀：`先评再改，优胜迭代。`

### 第 10 章：模型上下文协议（MCP）
- 章节定义：把工具和数据源按统一协议暴露给模型，让接入能力从“定制胶水”升级为“标准接口”。
- 核心组件：
  - MCP Server：暴露资源、工具、提示。
  - MCP Client：发现并调用远端能力。
  - 能力目录：资源 / 工具 / 提示三分法。
  - 传输层：STDIO、HTTP、SSE 等连接方式。
  - 访问控制：白名单、授权、过滤。
- 关键技术节点：
  - 动态发现“服务器会什么”。
  - 资源 / 工具 / 提示的边界区分。
  - 本地 STDIO 与远程 HTTP / SSE。
  - 工具过滤与最小可见集。
  - FastMCP 自动 schema 生成。
  - Agent 友好 API 设计。
- 典型落地：`统一接入文件系统、数据库、企业服务、表格与内容平台`。
- 风险边界：MCP 只解决“接得上”，不解决“底层 API 是否好用、是否安全、是否可维护”。
- 记忆口诀：`接口标准化，能力可发现。`

### 第 11 章：目标设定和监控
- 章节定义：把目标、成功标准和过程监控显式化，让 Agent 不只是“在忙”，而是“朝目标推进并可验证”。
- 核心组件：
  - 目标列表：明确系统要达成什么。
  - 成功标准：定义“完成”的判据。
  - 监控器：持续比较当前状态与目标差距。
  - Judge：评估是否达标。
  - 停止器：决定继续、修正还是结束。
- 关键技术节点：
  - 目标显式写入状态。
  - 成功 / 失败二值或分级判定。
  - 外部测试与指标化验收。
  - 里程碑与迭代次数上限。
  - executor / judge 角色分离。
  - 反馈轨迹存档。
- 典型落地：`代码生成闭环、项目里程碑推进、客服问题闭环处理`。
- 风险边界：同一个模型既写答案又判答案，很容易产生“自我放过”的错觉。
- 记忆口诀：`目标可验，过程可盯。`

## Part 3：可靠性、人在回路与知识接地（第 12-14 章）

### 第 12 章：异常处理和恢复
- 章节定义：让 Agent 在工具失败、超时、数据损坏或外部依赖失效时仍能识错、降级、恢复，而不是直接崩掉。
- 核心组件：
  - 异常探测器：识别错误、超时、脏输出。
  - 故障分类器：区分瞬时故障与持久故障。
  - 重试 / 回退器：尝试修复或切换备选路径。
  - 状态恢复器：清理坏状态并回到稳定点。
  - 升级器：把无法恢复的问题上抛给更高层或人工。
- 关键技术节点：
  - 超时、错误码、监控告警。
  - 瞬时错误重试策略。
  - fallback handler 兜底方案。
  - 优雅降级返回部分价值。
  - 状态回滚与脏数据清理。
  - 结构化异常日志。
- 典型落地：`主工具失败时切备用工具，数据库挂了就先返回缓存或简化答案`。
- 风险边界：如果只吞错不留痕，系统会长期静默失败而不自知。
- 记忆口诀：`先识错，再降级，最后升级。`

### 第 13 章：人机协同（HITL）
- 章节定义：在高风险、强模糊、强责任场景里，把人类判断纳入系统闭环，而不是追求全自动。
- 核心组件：
  - 升级规则：定义何时必须转人。
  - 人工工作台：查看上下文、证据和历史。
  - 证据包：给人类最小理解成本的摘要材料。
  - 人工裁决：给出最终批准、修改或驳回。
  - 反馈沉淀器：把人工修正变成后续优化素材。
- 关键技术节点：
  - human-in-the-loop 与 on-the-loop。
  - 风险分级与审批阈值。
  - 升级前自动摘要整理。
  - 脱敏与隐私保护。
  - 审批点 / 确认点插入工作流。
  - 人工反馈写回记忆或训练集。
- 典型落地：`法律 / 医疗 / 金融辅助、复杂客服、敏感内容审核、关键动作审批`。
- 风险边界：人越多，吞吐越低；若升级太晚，人接手时可能已经难以补救。
- 记忆口诀：`高风险交给人。`

### 第 14 章：知识检索（RAG）
- 章节定义：先检索外部知识，再基于证据生成答案，让模型从“闭卷作答”变成“开卷作答”。
- 核心组件：
  - 文档切块器：把知识拆成可检索单元。
  - 向量 / 混合检索器：找最相关证据。
  - 上下文注入器：把检索结果送入提示。
  - 生成器：根据证据组织最终答案。
  - 引用器：把依据和来源绑定到回答。
- 关键技术节点：
  - Embedding 与向量数据库。
  - BM25 + 语义检索混合策略。
  - 分块大小、重叠、元数据过滤。
  - rerank 与来源可信度判断。
  - GraphRAG 的关系增强。
  - Agentic RAG 的检索后推理。
- 典型落地：`企业知识问答、事实型客服、政策 / 法律检索、可溯源报告生成`。
- 风险边界：检索差时，生成再强也只是“基于错误上下文的流畅回答”。
- 记忆口诀：`先找证，再作答。`

## Part 4：协议、优化与高级智能（第 15-21 章）

### 第 15 章：Agent 间通信（A2A）
- 章节定义：让不同框架、不同服务、不同团队发布的 Agent 按统一协议彼此发现、调用和协作。
- 核心组件：
  - Agent Card：描述能力、端点、认证和用法。
  - 发现机制：找到可用 Agent。
  - 任务模型：提交、处理中、完成等生命周期。
  - 消息 / 工件模型：区分交流内容与最终产物。
  - 上下文关联器：用 contextId 串起多轮协作。
- 关键技术节点：
  - well-known URI / 注册表 / 直连发现。
  - 同步、异步、SSE、webhook 模式。
  - submitted / working / completed 状态机。
  - 远程认证与权限控制。
  - 工件流式返回与任务轮询。
  - A2A 与 MCP 的职责分离。
- 典型落地：`研究 Agent 调分析 Agent、企业多团队发布独立 Agent 服务`。
- 风险边界：协议统一不代表质量统一，远程 Agent 的真实可靠性仍需评估与监控。
- 记忆口诀：`Agent 找 Agent，按协议协作。`

### 第 16 章：资源感知优化
- 章节定义：让 Agent 在成本、时延、模型强度、上下文长度和可用算力之间动态平衡。
- 核心组件：
  - 复杂度分类器：判断问题值不值得“重兵出击”。
  - 模型路由器：给不同任务分配不同模型。
  - 预算管理器：控制 token、时延与花费。
  - 回退链：主模型失败时切备用路径。
  - 反馈器：根据质量和成本调整路由策略。
- 关键技术节点：
  - 简单 / 推理 / 搜索型请求分类。
  - 强模型与轻模型分工。
  - 上下文裁剪与摘要压缩。
  - 成本、token、时延统一观测。
  - 优雅降级与备用模型。
  - critic 对路由结果做复盘。
- 典型落地：`高价值任务走强模型，轻问答走便宜模型，最新事实问题自动接搜索`。
- 风险边界：若分类器误判，系统会“省了钱，却错了答案”。
- 记忆口诀：`难题配强模，简单走轻模。`

### 第 17 章：推理技术
- 章节定义：为 Agent 提供多步思考、分支探索、代码验证和外部行动结合的推理骨架。
- 核心组件：
  - 推理脚手架：规定思考步骤。
  - 分支搜索器：尝试多条候选路径。
  - 工具 / 代码验证器：用外部执行验证推理。
  - 反思循环：在错误后修正路径。
  - 推理预算器：限制轮次、时间和成本。
- 关键技术节点：
  - CoT：逐步链式思考。
  - ToT：多路径树搜索与回溯。
  - ReAct：思考-行动-观察回路。
  - PALM：程序辅助求解。
  - RLVR：可验证奖励强化。
  - MASS：多 Agent 系统搜索。
  - Deep Research：长链研究式推理。
- 典型落地：`复杂问答、数学证明、代码修复、研究分析、策略推演`。
- 风险边界：更长推理不等于更正确，很多时候只是把幻觉包装得更像“认真想过”。
- 记忆口诀：`想、试、验、收。`

### 第 18 章：Guardrails / 安全模式
- 章节定义：在输入、推理、工具调用和输出的全链路上加边界，让 Agent 既能做事，也不会越界做错事。
- 核心组件：
  - 输入守卫：过滤恶意请求和越狱尝试。
  - 策略执行器：判断是否合规。
  - 输出校验器：验证返回格式与内容边界。
  - 工具权限层：最小权限控制副作用。
  - 审计 / 人工接管通道：处理高风险情况。
- 关键技术节点：
  - prompt injection / jailbreak 检测。
  - 合规 JSON 输出。
  - Pydantic 或 schema 校验。
  - allow / deny / escalate 三态策略。
  - 工具白名单与最小权限。
  - 对抗样本回归测试。
- 典型落地：`客服合规回复、HR 去偏见、法律 / 科研场景风险拦截、危险操作阻断`。
- 风险边界：过严会误伤正常任务，过松则等于没防；只守输入不守工具和输出，会留下大洞。
- 记忆口诀：`先守边界，再放执行。`

### 第 19 章：评估和监控
- 章节定义：让 Agent 从“演示能跑”升级到“上线可观测、可度量、可复盘”。
- 核心组件：
  - 离线评估集：测固定任务质量。
  - 在线轨迹系统：记录真实运行过程。
  - Judge 模块：对主观质量给结构化评分。
  - 运营面板：看质量、时延、token、成本。
  - 漂移 / 异常告警器：发现行为变化。
- 关键技术节点：
  - 结果指标与轨迹指标双轨评估。
  - token / latency / error 监控。
  - LLM-as-a-Judge。
  - 合约式任务成功条件。
  - A/B 测试与基线对比。
  - 异常检测与漂移预警。
- 典型落地：`客服解决率看板、多 Agent 交接质量评估、线上任务质量抽检`。
- 风险边界：只看单一分数会诱导系统“优化指标”而非“优化真实价值”。
- 记忆口诀：`不上监控，不算上线。`

### 第 20 章：优先级排序
- 章节定义：当 Agent 同时面对多个任务、多个目标和多个约束时，决定“先做什么、后做什么”。
- 核心组件：
  - 任务模型：描述截止时间、影响、依赖与资源。
  - 评分规则：把任务优先级显式化。
  - 调度器：选择下一步执行项。
  - 重排器：在新事件出现时动态改顺序。
  - 防饥饿机制：避免低优任务永远得不到处理。
- 关键技术节点：
  - 紧急性 / 重要性 / 依赖打分。
  - 用户偏好权重。
  - 动态重新优先级排序。
  - 默认值与缺失信息填补。
  - 优先级变化原因日志。
  - 队列策略与任务老化。
- 典型落地：`项目管理、客服队列、网络安全告警、个人助理排程`。
- 风险边界：只看“急”不看“重”，系统会被噪音事件牵着走。
- 记忆口诀：`先急后重，还看依赖。`

### 第 21 章：探索和发现
- 章节定义：让 Agent 不只是执行既定流程，而是主动提出假设、设计实验、比较候选方向并拓展未知空间。
- 核心组件：
  - 假设生成器：提出新思路或研究方向。
  - 辩论 / 排名器：比较候选假设优劣。
  - 进化器：对高分方案改写、重组、扩展。
  - 实验回路：验证假设是否成立。
  - 人类科学家：负责方向校正与伦理把关。
- 关键技术节点：
  - 文献驱动的假设提出。
  - Elo 风格排名与竞争。
  - 相似假设聚类与接近度图。
  - 多 Agent 研究实验室结构。
  - 结果解释与外部证据验证。
  - 前置安全审查与伦理边界。
- 典型落地：`药物发现、算法搜索、开放式研究、漏洞探索、创意生成`。
- 风险边界：开放式探索极易把“新颖”误当“正确”，若无外部实验就只是语言层面的漂亮猜测。
- 记忆口诀：`先提假设，再做实验。`
