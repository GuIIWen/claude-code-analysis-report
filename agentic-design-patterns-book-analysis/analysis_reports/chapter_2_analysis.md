# 第 2 章：路由 (Routing) 分析报告

## 1. 核心定义
**路由 (Routing)** 是智能体（Agent）实现动态决策的核心机制。如果说提示词链是预设的“线性流水线”，那么路由就是流水线上的“分叉口”。它允许 Agent 根据环境状态、用户输入或上一步操作的结果，在多个潜在的行动路径（如不同的功能、工具或子流程）之间进行裁决和重定向，从而引入了条件逻辑。

## 2. 路由的四种实现方式
*   **基于 LLM 的路由**：利用大模型本身的推理能力，通过提示词要求模型对输入进行分类，并输出特定的路径标识符（如“订单查询”、“技术支持”）。
*   **基于嵌入 (Embedding) 的路由 (又称“语义路由”)**：将输入查询转换为向量，并与代表不同路径的向量进行语义相似度比较。这种方式适用于处理语义模糊但意图明确的场景。
*   **基于规则的路由**：使用关键词匹配、正则表达式或结构化数据判断（如 if-else 语句）。其优点是速度快、确定性强，但缺乏处理复杂语义的灵活性。
*   **基于专用 ML 模型的路由**：使用专门训练的分类器（判别式模型）进行路由。这种方式不依赖 LLM 的实时推理生成，而是将逻辑编码在微调模型的权重中，效率更高。

## 3. 典型应用场景
*   **意图识别与分发**：在虚拟助手或客服系统中，根据用户意图将请求路由到专门的子 Agent（如：投诉 -> 危机处理，咨询 -> 知识库）。
*   **自动化数据管线**：根据输入数据的格式（JSON/CSV）或元数据，将其定向到不同的转换函数或工作流。
*   **动态工具选择**：在拥有多个专业工具的系统中，充当“总调度器”，根据当前目标和系统的**累积状态 (Accumulated State)** 选择最合适的工具（如：数学题 -> 计算器，代码题 -> 编程助手）。

## 4. 关键组件与框架支持
*   **状态评估器**：执行评估并指导流程流向的机制。
*   **LangGraph 支持**：基于状态的图架构，非常适合根据系统的累积状态执行复杂的路由决策。
*   **Google ADK 支持**：通过定义子 Agent 的“能力”和“描述”，由框架内部逻辑（Auto-flow）自动匹配用户意图与功能。

## 5. 实践建议
*   **权衡灵活性与确定性**：对于简单的分类任务，优先考虑基于规则或嵌入的路由，以降低延迟和成本；只有在需要复杂推理时才使用 LLM 路由。
*   **兜底方案 (Fallback)**：必须设计处理“无法分类”或“意图不明确”情况的默认路由路径（如引导用户进一步澄清）。
*   **模块化子流程**：将路由到的每个分支设计为独立的、可测试的模块，以便于维护。

## 总结
第 2 章告诉我们，路由使 Agent 从“静态执行器”转变为“动态系统”。通过引入条件分支，Agent 能够超越确定的顺序处理，实现真正的自适应执行，灵活应对现实世界中多变的用户输入和复杂的运行状态。
