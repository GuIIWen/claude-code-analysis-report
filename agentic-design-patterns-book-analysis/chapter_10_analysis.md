第十章**“模型上下文协议”（Model Context Protocol / MCP）**是智能体生态标准化的重要里程碑。如果说工具使用（Tool Use）是让智能体学会“用手机”，那么 MCP 则是为所有手机应用定义了一套通用的“插头标准”，使得任何智能体都能即插即用地接入各种数据源和功能。

以下是对第十章内容的详细解读：

### 1. 核心定义：什么是 MCP？
MCP 是一个开放标准，旨在解决大语言模型（LLM）与外部工具/数据之间集成碎片化的问题。它采用“客户端-服务器”架构：MCP 服务器暴露资源和工具，MCP 客户端（智能体应用）动态发现并调用这些能力。这使得开发者无需为每个模型重复编写胶水代码。

### 2. MCP 的三大支柱
MCP 协议主要围绕以下三类元素展开：
- **资源 (Resources)**：模型可以读取的静态数据。例如：本地文件、数据库记录、API 返回的只读文档。
- **工具 (Tools)**：模型可以触发的可执行函数。例如：发送一条 Slack 消息、更新 Google 表格、运行一段 Python 脚本。
- **提示模板 (Prompts)**：预定义的交互模板。服务器可以向模型提供“如何使用这些资源”的最佳实践建议。

### 3. MCP 与普通函数调用的区别
| 特性 | 普通函数调用 (Custom Tooling) | 模型上下文协议 (MCP) |
| :--- | :--- | :--- |
| **复用性** | 强绑定于特定应用/框架 | 一次实现，所有支持 MCP 的模型通用 |
| **发现机制** | 硬编码在提示词或代码中 | 客户端动态询问：“你会做什么？” |
| **扩展性** | 每增加一个工具都要改代码 | 动态加载服务器，支持热插拔 |
| **传输层** | 通常是本地内存调用 | 支持标准输入输出 (STDIO) 和 HTTP/SSE (远程) |

### 4. 第十章中的典型案例与工具
- **文件系统访问**：通过 MCP 协议，智能体可以安全地列出目录、读取和写入本地文件，而无需开发者手动编写文件操作逻辑。
- **Google Sheets 集成**：将复杂的表格操作抽象为 MCP 工具，让模型能够像填表专家一样处理数据。
- **FastMCP**：这是一个 Python SDK，允许开发者用极其简洁的代码（只需几个装饰器）快速将现有的 Python 函数转换成 MCP 服务。
- **ADK Web**：Google ADK 提供的 Web 界面可以直接作为 MCP 客户端，方便开发者调试服务器逻辑。

### 5. 实践建议（来自书中总结）
- **改造 API 形状**：MCP 虽然能“连上”旧 API，但要让模型“用好”API，仍需将返回数据改造为模型友好的格式（如纯文本、Markdown 或干净的 JSON）。
- **传输方式选择**：本地工具推荐使用 `stdio`（简单、低延迟）；跨网络服务推荐使用 `HTTP/SSE`。
- **安全白名单**：在 MCP 工具集层设置 `tool_filter`，确保智能体只能看到当前任务确实需要的工具，防止误操作。
- **错误透明化**：在协议层返回详细的错误原因。如果模型收到的是模糊的“500 Error”，它将无法进行自我纠正或向用户解释原因。

### 6. 核心价值：生态互操作性
MCP 的真正威力在于它打破了供应商锁定。一个为 Claude 编写的 MCP 服务器，同样可以被 Gemini 或 GPT 的智能体调用。这标志着 Agent 开发正在从“烟囱式”开发向“协议驱动”开发转变。

**总结：**
第十章告诉我们，标准化是规模化的前提。MCP 为智能体提供了一个通用的“外设接口”，使得我们可以像组装电脑一样组装智能体系统。这是构建开放、可互操作的智能体生态系统的关键一步。
